## Spacing Scale

Declared values (must be multiples of 4). Source: Phase 1 UI-SPEC baseline — unchanged for Phase 2.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps (`gap-1`), badge internal padding (`py-0.5`), variable chip padding (`px-1`) |
| sm | 8px | Compact stacked spacing (`gap-2`), toolbar button spacing |
| md | 16px | Default element spacing (`p-4`, `gap-4`), form field gaps, editor body padding |
| lg | 24px | Section padding (`p-6`), campaign form section gaps |
| xl | 32px | Layout gaps (`p-8`), preview container padding |
| 2xl | 48px | Major section breaks (between form sections) |
| 3xl | 64px | Page-level top spacing (not used in this phase) |

Exceptions:
- TipTap editor minimum height: 320px — fixed, not a spacing token
- Preview email container max-width: 600px (`max-w-[600px]`) — fixed, not a spacing token
- Campaign form max-width on wide screens: unconstrained (full content area width like Phase 1 table)
- Touch targets for toolbar icon-only buttons: minimum 32px (Tailwind `w-8 h-8`)

## Typography

Source: Phase 1 UI-SPEC baseline — unchanged. Do not introduce new sizes or weights.

| Role | Size | Weight | Line Height | Tailwind class |
|------|------|--------|-------------|----------------|
| Body | 14px | 400 (regular) | 1.5 | `text-sm` |
| Label / Heading | 14–20px | 600 (semibold) | 1.2–1.4 | `text-sm font-semibold`, `text-base font-semibold`, `text-xl font-semibold` |

**Declared weights: 400 and 600 only.** Use `font-normal` (400) for form field body copy, placeholder text, editor body content, and status descriptions. Use `font-semibold` (600) for form section headings, field labels, page title, table column headers, and the campaign name in the list.

> **Footnote — brand mark exception:** The "MailOps" wordmark in the Sidebar uses `font-bold` (700). Pre-existing brand-mark token. Do NOT use weight 700 in any new component built for this phase.

Sizes used: 12px (`text-xs` — Badge, toolbar button tooltips, section sub-labels), 14px (`text-sm` — body, inputs, buttons, table rows), 16px (`text-base` — section headings, page-level headings), 20px (`text-xl` — page title). Do not introduce new sizes.

**Preview mode typography exception:** The preview container renders inside a light-theme (`bg-white text-gray-900`) email-like surface. Typography inside the preview is not constrained by the app's dark-theme typography — it renders the TipTap-generated HTML as-is, styled to approximate a real email client.

## Focal Point

**On `/campaigns` (list page):** The campaign list table is the primary visual anchor, with "New Campaign" as the dominant CTA in the page header. Empty state is the first thing a new user sees — it must be inviting, not clinical.

**On `/campaigns/new` and `/campaigns/:id/edit` (builder):** The TipTap editor is the primary visual focal point, occupying the widest element in the form. The "Schedule / Send" button at the bottom of the form is the terminal action — it must always be visible without excessive scrolling. The page header (campaign name + top-level action buttons) stays fixed context while the user works.

## Interaction Contracts

### Campaign List Page (`/campaigns`)

- Page header: `flex items-center justify-between` — page title "Campaigns" left, "New Campaign" `Button variant="primary" size="md"` right
- Layout: full-width table inside a `Card padding="sm"` — same pattern as contacts table
- Columns: Campaign name | Status (`Badge`) | Target list | Scheduled / Sent date | Actions (`...` menu)
- Column header: `text-xs font-semibold text-gray-400 uppercase tracking-wide bg-gray-900 border-b border-gray-800`
- Row height: 48px (`py-3`)
- Row hover: `hover:bg-gray-800/50`
- Clicking a campaign row (anywhere except the `...` menu) navigates to `/campaigns/:id/edit`
- Actions menu (`...` button, `Button variant="ghost" size="sm"`): "Edit", "Duplicate", "Delete" options
- "Duplicate" creates a copy with status `draft` and name prefixed "Copy of …" — triggers toast "Campaign duplicated."
- "Delete" shows inline confirmation (see Destructive Actions section)
- Pagination: 50 rows per page — Previous/Next `Button variant="secondary" size="sm"` below table, "Page N of N" indicator
- Empty state: centered within the card body — icon (`Mail` Lucide, `text-gray-600`, size 40px), heading, body copy, "New Campaign" `Button variant="primary" size="md"`

### Campaign Builder Form Layout (D-24, D-25)

- Route: `/campaigns/new` (new) and `/campaigns/:id/edit` (edit)
- Layout: single-page form — all sections visible on scroll, no wizard steps
- Page header: fixed top area with campaign name input (inline editable, prominent) left + action buttons right
- Campaign name: large `<input>` styled as `text-xl font-semibold text-gray-100 bg-transparent border-0 border-b border-gray-700 focus:border-indigo-500 focus:outline-none w-full py-1` — placeholder "Campaign name..."
- Action buttons (top-right, in a `flex gap-2`): "Save draft" `Button variant="secondary" size="sm"` + "Schedule / Send" `Button variant="primary" size="md"`
- Form body: `flex flex-col gap-8` — three labeled sections below the header
- Section label style: `text-xs font-semibold text-gray-400 uppercase tracking-wide mb-4`
- Sections in order: **1. Details** | **2. Content** | **3. Target & Schedule**

**Section 1 — Details:**
- Fields in a `grid grid-cols-1 md:grid-cols-2 gap-4`:
  - Subject line: `Input label="Subject line"` placeholder "Your email subject..." (supports `{{variable}}` typed directly)
  - Preview text: `Input label="Preview text"` placeholder "Short preview shown in inbox..."
  - Sender name: `Input label="Sender name"` placeholder "e.g. Alex from MailOps"
  - Sender email: `Input label="Sender email" type="email"` placeholder "noreply@yourdomain.com"

**Section 2 — Content:**
- Edit/Preview mode toggle bar above the editor: `flex gap-0 border border-gray-700 rounded-lg overflow-hidden w-fit`
  - "Edit" tab: `px-4 py-1.5 text-sm` — active: `bg-gray-800 text-indigo-400 border-b-2 border-indigo-500`, inactive: `bg-transparent text-gray-400 hover:text-gray-200`
  - "Preview" tab: same structure, toggles view
- **Edit mode:** TipTap editor (see Editor Toolbar section below)
- **Preview mode:** CampaignPreview component (see Preview section below)

**Section 3 — Target & Schedule:**
- Contact list selector: native `<select>` labeled "Target list" styled as `bg-gray-800 border border-gray-700 text-gray-100 text-sm rounded-lg px-3 py-2 w-full`; options populated from `useContactLists()` hook; first option "Select a contact list..." (disabled, default); each option shows list name + `(N contacts)`
- Scheduling: SchedulingSection component (see Scheduling section below)
- Test send: TestSendSection component (see Test Send section below)

- Toolbar button active (when format is active): add `bg-gray-700 text-gray-100` override; inactive: default ghost (`text-gray-400 hover:bg-gray-700 hover:text-gray-200`)
- Toolbar button groups separated by `<span className="w-px h-5 bg-gray-700 mx-1" />` (1px vertical divider)

**Toolbar button order and icons:**
1. Bold — `Bold` Lucide icon
2. Italic — `Italic` Lucide icon
3. [divider]
4. Heading 1 — `Heading1` Lucide icon
5. Heading 2 — `Heading2` Lucide icon

- Image popover: identical structure to link popover; URL input placeholder "https://..."; "Insert image" primary button

### Variable Insertion (D-05 to D-08)

**VariableDropdown:**
- Triggered by "Variable" toolbar button
- Dropdown panel: `absolute bg-gray-800 border border-gray-700 rounded-lg shadow-lg p-2 z-20 min-w-[180px]`
- Each variable option: `flex items-center gap-2 px-3 py-2 text-sm text-gray-200 hover:bg-gray-700 rounded-md cursor-pointer`
- Variable options (displayed label → inserted value):
  - "First name" → inserts `{{first_name}}` as VariableChipNode
  - "Last name" → inserts `{{last_name}}` as VariableChipNode
  - "Company" → inserts `{{company}}` as VariableChipNode
- Selecting a variable inserts the chip at current cursor position and closes the dropdown

**VariableChipNode (inline editor display):**
- Custom TipTap Node extension — inline, non-editable, atom (cannot place cursor inside)
- Visual: `inline-flex items-center bg-indigo-900 text-indigo-200 rounded px-1 text-sm font-mono mx-0.5 select-none`
- Displays the raw variable string: `{{first_name}}`, `{{last_name}}`, `{{company}}`
- Stored in the DB as the raw string (`{{first_name}}` — not the chip HTML) via `body_html` (HTML serialized by TipTap) and `body_json` (TipTap JSON state)
- Clicking a chip selects it as a node (TipTap default behavior) — Delete/Backspace removes it

**Variables in subject line (plain text `<input>`):**
- Subject line input does NOT use TipTap — it is a plain `<input type="text">`
- Users type `{{first_name}}` etc. directly — no chip rendering in the subject input
- Visual distinction via monospace hint only: subject input uses `font-mono` for the value when a variable string is detected — or simply documented that variables work but are not visually distinguished in the subject field (implementation discretion)

### Preview Mode (D-09 to D-11)

- Toggled by the Edit/Preview toggle bar above the editor
- CampaignPreview component replaces the TipTap editor (same bounding box)
- Preview container outer: `flex justify-center p-8 bg-gray-950 rounded-lg`
- Preview container inner: `max-w-[600px] w-full bg-white text-gray-900 p-8 rounded-lg shadow-xl`
- Before rendering: substitutes variables in `body_html`:
  - `{{first_name}}` → "Alex"
  - `{{last_name}}` → "Smith"
  - `{{company}}` → "Acme Corp"
- Substituted HTML is rendered inside the inner container via `dangerouslySetInnerHTML` — scoped inside the white preview panel (no dark-mode class inheritance)
- Sample data label: `text-xs text-gray-400 mb-2 text-center` shown above the preview container: "Preview with sample data: Alex Smith · Acme Corp"

### Scheduling Section (D-12 to D-15)

- Section within the Target & Schedule form section
- Label: "Delivery" — `text-xs font-semibold text-gray-400 uppercase tracking-wide mb-3`
- Two radio options in a `flex flex-col gap-3`:
  - "Send immediately" (default selected) — radio + label `text-sm text-gray-200`; description below label: `text-xs text-gray-400` "Campaign will be queued and sent right away."
  - "Schedule for later" — radio + label `text-sm text-gray-200`
- Radio button styled as native `<input type="radio">` — `accent-indigo-500` to color the native radio control in indigo
- When "Schedule for later" is selected, reveal (conditional `flex flex-col gap-3 mt-3 pl-6`):
  - Date/time: `<input type="datetime-local">` — styled `bg-gray-800 border border-gray-700 text-gray-100 text-sm rounded-lg px-3 py-2`; minimum value set to current datetime
  - Timezone: native `<select>` — same style as datetime input; populated from existing `TIMEZONES` constant
- Storage note (for executor): `scheduled_at` stored in UTC; timezone is display-only preference in the UI

### Test Send Section (D-16 to D-18)

- Placed below the Scheduling Section within Target & Schedule
- Label: "Test send" — `text-xs font-semibold text-gray-400 uppercase tracking-wide mb-3`
- Description: `text-sm text-gray-400 mb-3` "Send a test email to your account email address to verify content and formatting."
- Recipient display: `text-sm text-gray-300` — shows authenticated user's email address (from `useAuth()` profile)
- Button: "Send test email" `Button variant="secondary" size="sm" loading={testSending}` — with `Send` Lucide icon
- On success: `showToast("Test email sent.", "success")`
- On failure: `showToast("Test send failed. Check your Resend configuration.", "error")`
- No separate test send history UI — toast feedback only (D-18)

### Form Save and Submit Actions (D-26)

- Primary action row at bottom of the form: `flex justify-between items-center pt-6 border-t border-gray-800 mt-8`
- Left side: "Save draft" `Button variant="secondary" size="md"` — saves campaign with `status: 'draft'`
- Right side: "Schedule / Send" `Button variant="primary" size="md"` — validates required fields, then:
  - If "Send immediately" selected: sets `status: 'queued'`; transitions to Phase 3 delivery
  - If "Schedule for later" selected: sets `status: 'scheduled'`, saves `scheduled_at` in UTC
- Required fields for Schedule / Send: Campaign name, Subject line, Sender name, Sender email, Target list, email body (non-empty editor content)
- Validation failure: inline error text `text-sm text-red-400` appears below each missing required field; does not use toast for validation
- "Unsaved changes" guard: browser `beforeunload` event fires prompt if user navigates away with unsaved edits (D-23)

## Accessibility

- All interactive elements must have visible focus rings: `focus:outline-none focus:ring-2 focus:ring-indigo-500` (inherited from Phase 1 — apply consistently to all new inputs and buttons)
- TipTap editor container must have `aria-label="Email body editor"` and `role="textbox"` (TipTap handles these by default via ProseMirror; verify at implementation)
- Edit/Preview toggle: use `role="tablist"` on the container, `role="tab"` on each button, `aria-selected="true"` on the active tab
- VariableDropdown: `role="listbox"` on the dropdown panel, `role="option"` on each variable item
- VariableChipNode: `aria-label="{{variable_name}} — personalization variable"` on the chip span
- Campaign status badges must not rely on color alone — Badge text always included ("Draft", "Scheduled", "Sent", etc.)
- Scheduling radio buttons: standard `<input type="radio">` with associated `<label>` — no custom ARIA needed
- Form validation errors: use `aria-describedby` linking each input to its error message element; error element has `role="alert"`
- Delete confirmation (if inline): use `aria-live="polite"` on the confirmation region
- Campaign list row `...` menu button: `aria-label="Campaign options for {campaign_name}"`
- Link/Image popovers in toolbar: trap focus within the popover; Escape closes and returns focus to the toolbar button that opened it
