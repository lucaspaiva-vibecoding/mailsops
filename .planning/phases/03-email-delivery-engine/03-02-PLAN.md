**Route 1: GET /t/pixel/:trackingId** (per D-02, DELV-04)
- Extract `trackingId` from `c.req.param('trackingId')`
- Validate trackingId is a valid UUID format (regex: `/^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i`). If invalid, still return the pixel (don't break email rendering) but skip DB insert.
- Insert into `tracking_events`: `{ tracking_id: trackingId, event_type: 'open', occurred_at: new Date().toISOString(), user_agent: c.req.header('User-Agent') ?? null }`
- Check if this is the FIRST open for this tracking_id (query `tracking_events` for existing 'open' events with this tracking_id, count). If count === 0 (this is the first insert we just did, so check if count is 1 after insert — or use a simpler approach):
  - Use a subquery approach: after inserting the open event, check if the count of 'open' events for this tracking_id is exactly 1. If so, this is the first open — increment `campaigns.total_opened` via:
    1. Look up `campaign_id` from `campaign_recipients` WHERE `tracking_id = trackingId`
    2. Use Supabase RPC or raw SQL to increment: `await supabase.rpc('increment_counter', { row_id: campaignId, column_name: 'total_opened' })` — but since we don't have an RPC function, use the simpler pattern: fetch current value, increment, update. OR better: use a Postgres function.
  - **Simplest approach for MVP:** After inserting the open event, query `SELECT count(*) FROM tracking_events WHERE tracking_id = ? AND event_type = 'open'`. If count === 1, look up campaign_id from campaign_recipients and do `UPDATE campaigns SET total_opened = total_opened + 1 WHERE id = campaign_id`. Use raw SQL via `supabase.rpc()` or the update pattern.
  - **Recommended implementation:** Since Supabase JS doesn't support `total_opened + 1` directly, create a helper that:
    1. Gets campaign_id from campaign_recipients by tracking_id
    2. Gets current campaign row
    3. Updates with `total_opened: campaign.total_opened + 1`
  - Fire-and-forget the DB operations (don't await before returning pixel). Use `c.executionCtx.waitUntil()` if available in Hono, otherwise just don't block the response — but in Deno Edge Functions we should await to ensure writes complete. Await the insert but return quickly.
- Return the pixel PNG:
  ```typescript
  return new Response(PIXEL_BYTES, {
    headers: {
      'Content-Type': 'image/png',
      'Cache-Control': 'no-store, no-cache, must-revalidate',
      'Pragma': 'no-cache',
    },
  })
  ```

**Route 2: GET /t/click/:trackingId/:linkIndex** (per D-03, DELV-05)
- Extract `trackingId` and `linkIndex` from params
- Validate trackingId UUID format. If invalid, return 400 plain text.
- Look up `campaign_recipients` row by `tracking_id = trackingId` to get `link_map` and `campaign_id`
- Get original URL: `const originalUrl = data?.link_map?.[linkIndex]`
- If no recipient found or no URL in link_map for that index, redirect to a safe fallback (return 400 "Invalid tracking link")
- **Security: Only redirect to URLs stored in the DB link_map** — never accept a URL from query params (prevents open redirect)
- Insert into `tracking_events`: `{ tracking_id: trackingId, event_type: 'click', link_index: parseInt(linkIndex, 10), link_url: originalUrl, occurred_at: new Date().toISOString(), user_agent: c.req.header('User-Agent') ?? null }`
- Increment `campaigns.total_clicked`: fetch campaign row, update `total_clicked: campaign.total_clicked + 1`
- Return `Response.redirect(originalUrl, 302)`

**Route 3: GET /t/unsub/:trackingId** (per D-04, DELV-06)
- Extract `trackingId` from params
- Validate trackingId UUID format. If invalid, return 400 HTML.
- Look up `campaign_recipients` row by `tracking_id = trackingId` to get `contact_id` and `campaign_id`
- If no recipient found, return HTML page: "Invalid unsubscribe link."
- Update `contacts` table: `{ status: 'unsubscribed', unsubscribed_at: new Date().toISOString() }` WHERE `id = contact_id`
- Insert tracking event: `{ tracking_id: trackingId, event_type: 'unsubscribe' }` — note: we use 'unsubscribe' as event_type even though it's not in the TrackingEvent union (tracking_events table allows any TEXT)
- Increment `campaigns.total_unsubscribed`: fetch campaign, update `total_unsubscribed: campaign.total_unsubscribed + 1`
- Return HTML confirmation page (per D-04 — render confirmation, not just redirect):

**Helper function for incrementing campaign counters:**
```typescript
async function incrementCampaignCounter(campaignId: string, field: string) {
  const { data: campaign } = await supabase
    .from('campaigns')
    .select(field)
    .eq('id', campaignId)
    .single()
  if (campaign) {
    await supabase
      .from('campaigns')
      .update({ [field]: (campaign as Record<string, number>)[field] + 1 })
      .eq('id', campaignId)
  }
}
```

**Helper function for looking up campaign_id from tracking_id:**
```typescript
async function getRecipient(trackingId: string) {
  const { data } = await supabase
    .from('campaign_recipients')
    .select('contact_id, campaign_id, link_map')
    .eq('tracking_id', trackingId)
    .single()
  return data
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Email client -> /t/* endpoints | Anonymous HTTP requests from email clients (no auth) |
| /t/click -> redirect URL | Click redirect could be an open-redirect vector if URL comes from user input |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Spoofing | /t/pixel/:trackingId | accept | Tracking IDs are UUID v4 (128-bit random) — computationally infeasible to guess; pixel returns same PNG regardless of validity |
| T-03-05 | Tampering | /t/unsub/:trackingId | mitigate | UUID tracking_id makes mass unsubscribe via enumeration infeasible; each ID uniquely maps to one contact per campaign |
| T-03-06 | Information Disclosure | /t/* error responses | mitigate | Invalid tracking IDs return generic responses (pixel PNG or "Invalid" HTML) — no internal state leaked |
| T-03-07 | Denial of Service | /t/pixel rate | accept | Supabase Edge Functions have built-in rate limiting; pixel endpoint is stateless and lightweight; no amplification vector |
| T-03-08 | Elevation of Privilege | /t/click open redirect | mitigate | Click redirect only uses URLs from the DB `link_map` column — never from query params or request body; prevents open redirect attacks |
</threat_model>
